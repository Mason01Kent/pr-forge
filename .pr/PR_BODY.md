## Summary

This PR prepares the final release bundle for PR Forge v1.6.10. It archives all historical VSIX artifacts (1.2.3–1.6.9) into an `OldVersions/` directory, updates the AI provider model to `deepseek-v4-pro`, and includes the generated PR metadata files that were produced by the extension itself.

## Why this matters

- **Release completeness:** Keeping past VSIX files in the repository preserves a full release history and simplifies artifact retrieval for users or CI.
- **Model upgrade:** Switching to `deepseek-v4-pro` keeps the project aligned with the latest DeepSeek model capabilities.
- **Dogfooding:** The generated `.pr/PR_BODY.md`, `.pr/PR_REVIEW.md`, and `.pr/PR_TITLE.txt` files demonstrate the extension’s output and serve as a record of the current PR generation quality.

## Changes

- **Config:** Updated `defaultModel` in `.pr-forge.json` from `deepseek-chat` → `deepseek-v4-pro`.
- **Archives:** Added `OldVersions/` directory containing 21 VSIX binaries from v1.2.3 through v1.6.9.
- **Generated metadata:**
  - `.pr/PR_BODY.md` – Full PR description generated by PR Forge (summary, why, changes, tests, review focus, risks).
  - `.pr/PR_REVIEW.md` – Structured code review with overall assessment, blocking issues, suggestions, security concerns, test coverage, and recommendation.
  - `.pr/PR_TITLE.txt` – Suggested PR title: “Add initial project structure and files”.
  - `.pr/PR_BODY.state.json` – State snapshot (branch, base, head SHA, timestamp).

## Tests / verification

All **100 tests pass** (0 failures, 0 errors). Tested areas include:

- Schema upgrade paths (v1→v8, v5→v8, v7→v8)
- Fresh config defaults and field preservation
- SCM providers (GitHub and GitLab PR/MR operations, labels, reviews, merge readiness)
- Remote URL parsing (HTTPS, SSH, enterprise, self-managed)
- Model limits, model listing, and provider integration
- Diff batching, annotation, and review comment mapping
- Markdown rendering, template discovery, telemetry helpers, and submit flow utilities

## Review focus

1. **VSIX archive integrity** – Confirm all binary files are the correct historical releases and are not corrupted.
2. **Model change** – Verify that `deepseek-v4-pro` is a valid, supported model and does not introduce breaking changes (e.g., token limits, API compatibility).
3. **Generated PR files** – Check that the included `.pr/` files reflect the actual state of this branch and are not stale or misleading.

## Risks / follow-ups

- **Binary files** – The VSIX archives increase repository size; consider adding them to `.gitattributes` for LFS if not already done.
- **Generated PR metadata** – These files will be regenerated on future runs and may cause conflicts if manually edited. They are safe as a snapshot, but future automation should handle them gracefully.
- **Model migration** – If any code relies on hardcoded model names, ensure `deepseek-v4-pro` is recognized by the provider configuration and token budget logic.
- **No code logic changes** – The PR contains no source code modifications, only configuration, archives, and tooling outputs; deployment risk is minimal.
